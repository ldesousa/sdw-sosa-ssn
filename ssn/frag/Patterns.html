Common Modeling Questions

The SSN ontology provides a set of general components for describing Actuations, Observations and Samplings. However, the supporting classes (i.e. Procedure, ActuationProcedure, ObservationProcedure, SamplingProcedure, System, Actuator, Sensor, Sampler, FeatureOfInterest, Sample, Property, ObservableProperty etc) have no properties specified except for the relationships between the classes. Furthermore, there are a large number of these relation predicates, which can lead to a rather tangled dataset in practice. Thus, any application that uses SSN must select an implementation pattern to constrain the variability. And in order to support interoperability with other applications, the implementation pattern should be formalized and shared.

It is beyond the scope of this document to provide implementation profiles. However, a number of frequent requirements are discussed and illustrated in this section, including locations, forecasts, and quantity values with a unit of measure.

OGC is planning to develop a register of SSN implementation profiles to support the development of interoparable communities using SSN.

Location, sampling location

Many of the key classes provided by SOSA and SSN represent entities that can be located in space such as sensors, features of interest, actuators, samples, and so forth, or activities that can be located via their participating entities, e.g., platforms. These entities will usually be described using models and ontologies defined for application domains, including technical disciplines, social and business contexts.

There are a number of standard implementations that support the expression of spatial properties, including location. These are discussed further in the Spatial Data on the Web Best Practices note [[SDW-BP]]. In particular, GeoSPARQL [[geosparql]] provides a flexible and relatively complete platform for geospatial objects, that fosters interoperability between geo-datasets.

Using GeoSPARQL, geometries can be assigned to entities using the geo:hasGeometry property. For example, if observations are made of the atmosphere at a specific location, it is recommended that this be defined as a Sample whose location is given using GeoSPARQL with the following pattern:

Generic or Specific Instances of sosa:System

    
This specification does not specify whether an instance of sosa:System
        should be generic (e.g., ex:TemperatureSensor,
        ex:LightActuator), or specific to a single feature of interest (e.g.,
        <temperatureSensor/84>, <light/112>).
        Implementers are free to choose one way of modeling things or the other.
